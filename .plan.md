# RainMusic 三端代码优化与UI美化计划

## Context

RainMusic 是一个本地音乐播放器项目，包含 iOS (SwiftUI)、Android (Jetpack Compose)、HarmonyOS (ArkTS) 三个平台版本。经过全面代码审查，发现三端存在多处关键 Bug（内存泄漏、响应式机制损坏、类型错误等）、代码质量问题（大量重复代码、架构不合理、性能瓶颈）以及 UI 与设计稿不一致的情况。本计划旨在系统性修复并优化，使三端代码质量统一提升，UI 对齐 `UI/music-player/index.html` 设计稿的海军蓝深色主题风格。

---

## Phase 1: 关键 Bug 修复

### Task 1.1 iOS — Timer 泄漏修复
- **文件**: `RainMusic/Views/PlayerView.swift`
- **问题**: `onAppear` 中创建的 `Timer.scheduledTimer` 永不停止，View 消失后持续运行
- **修复**: 用 `@State` 存储 Timer 引用，`onDisappear` 中 `invalidate()`

### Task 1.2 iOS — MPVolumeView 泄漏修复
- **文件**: `RainMusic/Managers/AudioPlayerManager.swift`
- **问题**: `MPVolumeView` 添加到 window 后从未移除
- **修复**: 保留强引用，提供 cleanup 方法移除

### Task 1.3 iOS — MiniPlayerView 进度条不更新
- **文件**: `RainMusic/Views/MiniPlayerView.swift`
- **问题**: 无刷新驱动，进度条静止
- **修复**: 使用 `TimelineView(.periodic(from: .now, by: 0.5))` 包裹进度条

### Task 1.4 Android — Handler 泄漏修复
- **文件**: `RainMusicAndroid/.../manager/AudioPlayerManager.kt`
- **问题**: 进度更新和音量检查的 Handler/Runnable 永不停止
- **修复**: 存为成员变量，在 `release()` 中 `removeCallbacksAndMessages(null)`；仅在播放时启动

### Task 1.5 Android — SongDao.deleteById 类型修复
- **文件**: `RainMusicAndroid/.../data/db/SongDao.kt`
- **问题**: `deleteById(id: Int)` 应为 `Long`
- **修复**: 参数类型改为 `Long`

### Task 1.6 Android — themes.xml 白闪修复
- **文件**: `RainMusicAndroid/.../res/values/themes.xml`
- **问题**: 使用 Light 父主题，冷启动闪白
- **修复**: 改为 `Material.NoActionBar`，设 `windowBackground` 为 `#0A1628`

### Task 1.7 Android — 删除死代码 MiniPlayer.kt
- **文件**: `RainMusicAndroid/.../ui/component/MiniPlayer.kt`
- **问题**: 166 行代码从未被调用
- **修复**: 删除文件（BottomTabBar 已内联实现迷你播放）

### Task 1.8 Android — 启用 Release 混淆 + 清理未用依赖
- **文件**: `RainMusicAndroid/app/build.gradle.kts`
- **修复**: `isMinifyEnabled = true`；删除未使用的 `navigation-compose` 和 `accompanist-permissions` 依赖

### Task 1.9 HarmonyOS — ViewModel 响应式机制重构
- **文件**: `RainMusicHarmony/.../viewmodel/PlayerViewModel.ets`
- **问题**: `stateVersion++` + `songs = [...this.songs]` 反模式，getter 代理无法触发响应式
- **修复**: 改为直接属性存储，使用 `@Track` 装饰器精确追踪

### Task 1.10 HarmonyOS — BottomTabBar ViewModel 实例修复
- **文件**: `RainMusicHarmony/.../components/BottomTabBar.ets`
- **问题**: 组件内 `new PlayerViewModel()` 创建独立实例，与主页面状态不同步
- **修复**: 通过 `@Link` 从父组件传入 ViewModel

### Task 1.11 HarmonyOS — 音量轮询改事件监听 + 去除重复进度更新
- **文件**: `RainMusicHarmony/.../manager/AudioPlayerManager.ets`
- **问题**: `setInterval` 500ms 轮询音量（应用事件监听）；进度同时用 `setInterval` 和 `timeUpdate` 更新（重复）
- **修复**: 音量改用 `volumeGroupManager.on('volumeChange')`；移除进度 `setInterval`，仅保留 `timeUpdate`

## Phase 2: 代码架构与质量改进

### Task 2.1 iOS — PlayerView 横竖屏重复代码消除
- **文件**: `RainMusic/Views/PlayerView.swift`
- **问题**: 横竖屏 200+ 行重复代码（封面、进度条、控制面板）
- **修复**: 提取 `ProgressBarView`、`ControlPanelView`、`SongInfoView` 为独立 `@ViewBuilder`，横竖屏仅保留布局差异

### Task 2.2 iOS — PlayerViewModel 简化
- **文件**: `RainMusic/ViewModels/PlayerViewModel.swift` + `RainMusic/Views/PlayerView.swift`
- **问题**: ViewModel 是 AudioManager 的薄包装，几乎所有方法透传
- **修复**: 将歌词管理逻辑移入 AudioPlayerManager，删除 PlayerViewModel，View 直接使用 AudioManager

### Task 2.3 iOS — 数据库扫描性能优化
- **文件**: `RainMusic/Managers/MusicLibraryManager.swift`
- **问题**: 扫描时每首歌单独 fetch 去重（N+1 查询）；封面存 PNG 体积大
- **修复**: 扫描前批量加载已有 ID 到 `Set`；封面改 `jpegData(compressionQuality: 0.85)`

### Task 2.4 Android — PlayerScreen 拆分 + 重复代码消除
- **文件**: `RainMusicAndroid/.../ui/screen/PlayerScreen.kt`
- **问题**: 720 行单文件含 9 个 Composable；竖屏封面与 CoverSection 重复
- **修复**: 拆分为 `player/` 子目录多文件；竖屏复用 CoverSection

### Task 2.5 Android — PlayerViewModel 职责拆分
- **文件**: `RainMusicAndroid/.../viewmodel/PlayerViewModel.kt`
- **问题**: 207 行管理播放+扫描+导入+排序+搜索+歌词+封面
- **修复**: 拆分为 `PlayerViewModel`（播放控制+歌词）+ `LibraryViewModel`（音乐库管理）

### Task 2.6 HarmonyOS — 常量提取 + 颜色统一管理
- **新建**: `RainMusicHarmony/.../constants/Colors.ets`, `constants/Layout.ets`
- **问题**: 颜色 hex 散落各文件；大量魔法数字
- **修复**: 创建统一颜色和布局常量文件，全项目引用

### Task 2.7 HarmonyOS — 数据持久化
- **文件**: `RainMusicHarmony/.../manager/MusicScanner.ets` + 新建 `data/SongDatabase.ets`
- **问题**: 歌曲列表无持久化，重启丢失
- **修复**: 使用 `@ohos.data.relationalStore` 实现歌曲数据持久化

## Phase 3: UI 设计稿对齐

### Task 3.1 底部 Tab 栏选中态颜色修正
- **设计稿规格**: 选中态背景 `#4FC3F7` (accent)，文字 `#0A1628` (bg-dark)
- **涉及文件**:
  - Android: `BottomTabBar.kt` — 当前选中态为 BgBorder 半透明背景 + accent 文字，需修正
  - iOS: `BottomTabBarView.swift` — 校验并对齐
  - HarmonyOS: `BottomTabBar.ets` — 校验并对齐

### Task 3.2 HarmonyOS — Emoji 替换为系统图标
- **涉及文件**: `PlayerPage.ets`, `BottomTabBar.ets`, `Index.ets`
- **修复**: 所有 Emoji (🎵▶⏸⏭⇄↻🎤🎶📥) 替换为 HarmonyOS 系统符号资源

### Task 3.3 Now Playing 屏幕间距对齐
- **设计稿规格**: 封面 280px, 标题 28px bold, 进度条 4px, 控制面板 80px/16px radius, 播放按钮 56px 圆形
- **涉及文件**:
  - HarmonyOS: `PlayerPage.ets` — 封面 260→280vp
  - 三端统一进度条样式（细轨道 + 拖拽圆点）

### Task 3.4 歌曲行样式统一
- **设计稿规格**: 封面 48px/6px radius, 标题 15px/600, 艺术家 12px, 时长 13px, 行高 64px
- **涉及文件**: 三端的 `SongRowView` / `SongRow` 组件对齐设计稿尺寸

## Phase 4: 打磨与体验优化

### Task 4.1 三端无障碍支持
- iOS: 所有按钮添加 `.accessibilityLabel()`，进度条添加 `.accessibilityValue()`
- Android: Icon 补充 `contentDescription`，进度条添加 `semantics`
- HarmonyOS: 所有交互组件添加 `.accessibilityText()`

### Task 4.2 iOS — 手势冲突修复
- **文件**: `RainMusic/Views/PlayerView.swift`
- **修复**: 切歌手势改用 `highPriorityGesture` 或限定触发区域，避免与 ScrollView 冲突

### Task 4.3 HarmonyOS — 进度条动态宽度
- **文件**: `RainMusicHarmony/.../pages/PlayerPage.ets`
- **修复**: 用 `onAreaChange` 获取实际宽度替代硬编码 `300`

### Task 4.4 Android — Material You 动态颜色
- **文件**: `RainMusicAndroid/.../ui/theme/Theme.kt`
- **修复**: API 31+ 支持 `dynamicDarkColorScheme()`，保留自定义深色作为 fallback

## 验证方案

1. **Phase 1 验证**: 每修复一个 Bug 后运行 App 验证功能正常；iOS 用 Instruments 检测泄漏；Android 用 LeakCanary 检测
2. **Phase 2 验证**: 重构后运行全部功能（播放、切歌、扫描、导入、歌词）确认无回归
3. **Phase 3 验证**: 与设计稿 HTML 并排对比截图，检查颜色、间距、图标一致性
4. **Phase 4 验证**: 开启各平台无障碍功能（VoiceOver/TalkBack/屏幕阅读器）遍历所有界面
